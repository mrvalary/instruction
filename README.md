## Введение
**Git** — это распределённая система контроля версий. Она позволяет:
- сохранять историю изменений проекта
- работать с ветками
- объединять изменения
- возвращаться к предыдущим версиям файлов
- работать над одним проектом в команде

**GitHub** — это онлайн-сервис для хранения Git-репозиториев и совместной работы над проектами.

### Зачем нужен Git
- **Контроль версий** — вы всегда знаете, кто, когда и зачем внёс изменения
- **Резервное копирование** — код хранится не только на вашем компьютере
- **Совместная работа** — несколько человек могут работать над одним проектом
- **Ветвление** — можно экспериментировать, не ломая основную версию
- **История** — можно откатиться к любому предыдущему состоянию проекта

### Основные понятия

| Термин | Описание |
|--------|----------|
| Репозиторий | Папка проекта, за которой следит Git |
| Коммит | Сохранённое состояние проекта |
| Ветка | Независимая линия разработки |
| Индекс | Область, где подготавливаются файлы перед коммитом |
| Удалённый репозиторий | Копия проекта на сервере (например, GitHub) |
| Клонирование | Создание локальной копии удалённого репозитория |
| Fork | Копия чужого репозитория в вашем аккаунте |

### Как работает Git (кратко)

1. Вы создаёте репозиторий
2. Вносите изменения в файлы
3. Добавляете изменения в индекс (git add)
4. Сохраняете изменения в виде коммита (git commit)
5. Отправляете коммиты на GitHub (git push)
6. Другие участники скачивают изменения (git pull)

### Что такое Markdown

**Markdown** — это облегчённый язык разметки. Файлы с расширением `.md` используются для документации на GitHub. Данная инструкция написана именно на Markdown.
Примеры синтаксиса Markdown:
- Заголовки создаются с помощью символа #
- Списки создаются с помощью дефиса или цифр
- Код выделяется тройными обратными кавычками
- Ссылки оформляются в квадратных и круглых скобках
- Таблицы создаются с помощью вертикальной черты

## Локальная работа с Git

Локальный Git используется для работы с репозиторием на компьютере пользователя.

### Инициализация репозитория

Команда создаёт новый локальный Git-репозиторий.
``` bash
git init
```

### Проверка состояния репозитория

Команда показывает изменённые, добавленные и неотслеживаемые файлы.
``` bash
git status
```

### Добавление файла в индекс

Команда добавляет файл в область подготовленных изменений.
``` bash
git add имя_файла
```
Пример:
``` bash
git add README.md
```

### Добавление всех файлов

Команда добавляет все изменённые файлы в индекс.
``` bash
git add .
```

### Создание коммита

Коммит фиксирует изменения в истории репозитория.
``` bash
git commit -m "Сообщение коммита"
```
Пример:
``` bash
git commit -m "Добавлена инструкция по Git"
```

### Просмотр истории коммитов

Команда показывает историю изменений.
``` bash
git log
```

### Ключи команды git log

| Ключ | Назначение |
|------|------------|
| --oneline | Каждый коммит в одну строку |
| --graph | История в виде графа веток |
| --all | Коммиты из всех веток |
| --stat | Краткая статистика изменений |
| -p | Подробные изменения в файлах |
| -n 5 | Последние 5 коммитов |
| --author="Имя" | Коммиты конкретного автора |
| --since="2024-01-01" | Коммиты после указанной даты |
| --until="2024-12-31" | Коммиты до указанной даты |
| --grep="текст" | Поиск по тексту сообщения |

Пример:
``` bash
git log --oneline --graph --all
```

### Просмотр изменений

Показывает изменения в файлах, которые ещё не добавлены в индекс.
``` bash
git diff
```
Показывает изменения, которые уже добавлены через git add.
``` bash
git diff --staged
```

### Удаление файлов

Удаляет файл и сразу добавляет это изменение в индекс.
``` bash
git rm file.txt
```
Перестаёт отслеживать файл, но оставляет его на диске.
``` bash
git rm --cached file.txt
```

### Переименование файлов
``` bash
git mv old-name.txt new-name.txt
```

### Отмена изменений

Отменяет незакоммиченные изменения в файле.
``` bash
git restore file.txt
```
Убирает файл из индекса, но сохраняет изменения.
``` bash
git restore --staged file.txt
```

### Создание ветки

Команда создаёт новую ветку.
``` bash
git branch имя_ветки
```
Пример:
``` bash
git branch local
```

### Переключение между ветками

Команда переключает пользователя на другую ветку.
``` bash
git switch имя_ветки
```
Пример:
``` bash
git switch local
```
Альтернативная команда:
``` bash
git checkout local
```

### Просмотр списка веток

Список локальных веток.
``` bash
git branch
```
Список всех веток, включая удалённые.
``` bash
git branch -a
```

### Удаление ветки

Удаляет ветку, если она уже слита.
``` bash
git branch -d branch-name
```
Принудительное удаление.
``` bash
git branch -D branch-name
```
### Временное сохранение изменений

Прячет текущие изменения.
``` bash
git stash
```
Возвращает спрятанные изменения.
``` bash
git stash pop
```

### Пример полного цикла локальной работы
``` bash
git init
git add .
git commit -m "Первый коммит"
git branch develop
git switch develop
echo "Новая функция" > feature.txt
git add feature.txt
git commit -m "Добавлена новая функция"
git switch main
git merge develop
```

## Работа с удалённым репозиторием

Удалённый репозиторий хранится на GitHub или другом Git-сервисе.

### Добавление удалённого репозитория

Команда связывает локальный репозиторий с удалённым.
``` bash
git remote add origin ссылка_на_репозиторий
```
Пример:
``` bash
git remote add origin https://github.com/username/git-instruction.git
```

### Проверка удалённого репозитория

Команда показывает список подключённых удалённых репозиториев.
``` bash
git remote -v
```

### Удаление удалённого репозитория
``` bash
git remote remove origin
```

### Отправка изменений на GitHub

Команда отправляет локальные коммиты в удалённый репозиторий.
``` bash
git push origin имя_ветки
```
Пример:
``` bash
git push origin main
```

### Первая отправка ветки

Команда отправляет ветку и связывает её с удалённой веткой.
``` bash
git push -u origin имя_ветки
```
Пример:
``` bash
git push -u origin main
```
После этого можно писать просто git push.

### Получение изменений из удалённого репозитория

Команда скачивает изменения и сразу объединяет их с текущей веткой.
``` bash
git pull origin имя_ветки
```
Пример:
``` bash
git pull origin main
```

### Загрузка изменений без слияния

Команда скачивает информацию об изменениях, но не объединяет их автоматически.
``` bash
git fetch
git fetch --all
```

### Клонирование репозитория

Команда создаёт локальную копию удалённого репозитория.
``` bash
git clone ссылка_на_репозиторий
```
Пример:
``` bash
git clone https://github.com/username/git-instruction.git
```

### Удаление ветки на удалённом репозитории
``` bash
git push origin --delete branch-name
```

### Работа с тегами

Создание тега:
``` bash
git tag v1.0
```
Создание тега с описанием:
``` bash
git tag -a v1.0 -m "Релиз версии 1.0"
```
Отправка тегов на GitHub:
``` bash
git push --tags
```

## Работа с форками

**Fork** — это копия чужого репозитория в вашем аккаунте GitHub.
Форки используются, когда нужно предложить изменения в чужой проект.

### Как сделать fork

1. Открыть чужой репозиторий на GitHub
2. Нажать кнопку **Fork**
3. GitHub создаст копию репозитория в вашем аккаунте

### Клонирование форка
``` bash
git clone ссылка_на_форк
```
Пример:
``` bash
git clone https://github.com/username/project-fork.git
```

### Добавление исходного репозитория

Команда добавляет оригинальный репозиторий как upstream.
``` bash
git remote add upstream ссылка_на_исходный_репозиторий
```
Пример:
``` bash
git remote add upstream https://github.com/original-owner/project.git
```

### Получение изменений из исходного репозитория
``` bash
git fetch upstream
```

### Обновление своей ветки из upstream
``` bash
git merge upstream/main
```

### Полный цикл работы с fork

1. Сделать fork репозитория на GitHub
2. Клонировать свой fork:
``` bash
git clone https://github.com/username/repo.git
cd repo
```
3. Добавить upstream:
``` bash
git remote add upstream https://github.com/original/repo.git
```
4. Создать ветку для изменений:
``` bash
git branch my-changes
git switch my-changes
```
5. Внести изменения, закоммитить:
``` bash
git add .
git commit -m "Мои изменения"
```
6. Отправить в свой fork:
``` bash
git push -u origin my-changes
```
7. На GitHub нажать **New Pull Request**
8. Выбрать исходную и целевую ветки
9. Описать изменения и отправить

### Синхронизация fork с оригиналом
``` bash
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
```
## Слияние веток

Слияние используется, чтобы объединить изменения из одной ветки в другую.

### Базовое слияние
``` bash
git checkout main
git merge local
```
### Возможные конфликты

Если одни и те же строки изменены в обеих ветках, Git сообщит о конфликте.

1. Открыть файлы с конфликтами
2. Выбрать правильный вариант вручную
3. Выполнить:
``` bash
git add .
git commit -m "Конфликт разрешён"
```
### Отмена слияния
``` bash
git merge --abort
```

## Частые ошибки и их решение

### 1. Забыли добавить файлы перед коммитом

Ошибка: выполнили `git commit`, но изменения не сохранились.
Причина: не выполнили `git add`.
Решение:
```bash
git add .
git commit -m "Сообщение"
```

### 2. Сообщение коммита пустое

Ошибка: `Aborting commit due to empty commit message`
Решение: указывайте сообщение в кавычках:
```bash
git commit -m "Исправлен баг с авторизацией"
```

### 3. Изменения не отправляются на GitHub

Ошибка: `failed to push some refs`
Причина: в удалённом репозитории есть изменения, которых нет локально.
Решение:
```bash
git pull origin main
git push origin main
```

### 4. Конфликт слияния

Ошибка: `CONFLICT (content): Merge conflict in file.txt`
Решение:
1. Открыть файл с конфликтом
2. Удалить маркеры `<<<<<<<`, `=======`, `>>>>>>>`
3. Оставить нужный код
4. Выполнить:
```bash
git add .
git commit -m "Конфликт разрешён"
```

### 5. Закоммитили не в ту ветку

Решение:
```bash
git log --oneline           # найти хеш нужного коммита
git cherry-pick <хеш>       # перенести коммит в текущую ветку
git checkout старая-ветка
git reset --hard HEAD~1     # удалить коммит из старой ветки
```

### 6. Хотим отменить последний коммит

Если коммит ещё не отправлен на GitHub:
```bash
git reset --soft HEAD~1
```
Изменения останутся в индексе, можно поправить и перекоммитить.

Если коммит уже отправлен:
```bash
git revert HEAD
```
Создаст новый коммит, отменяющий изменения.

### 7. Переключились не на ту ветку

Решение:
```bash
git switch -
```

### 8. Случайно удалили файл

Если файл был в последнем коммите:
```bash
git restore file.txt
```

Если файл был удалён давно:
```bash
git log -- file.txt          # найти коммит, где файл ещё был
git checkout <хеш> -- file.txt
```

### 9. Клонировали репозиторий и не можете запушить

Ошибка: `Permission denied`
Причина: это чужой репозиторий, у вас нет прав на запись.
Решение:
1. Сделать fork репозитория
2. Клонировать свой fork
3. Вносить изменения в него
4. Создать Pull Request

### 10. Забыли переключиться с main и работали в ней

Решение:
```bash
git stash                    # спрятать изменения
git branch new-branch        # создать новую ветку
git switch new-branch        # переключиться
git stash pop                # вернуть изменения
git add .
git commit -m "Изменения в новой ветке"
```

### 11. Закоммитили файлы, которые не должны быть в репозитории

Решение: создайте файл `.gitignore`:
```bash
node_modules/
.env
*.log
```
Затем:
```bash
git rm -r --cached node_modules
git add .gitignore
git commit -m "Добавлен gitignore"
```

### 12. Хотим посмотреть, что изменилось в последнем коммите

```bash
git show HEAD
```
Или посмотреть изменения в конкретном файле:
```bash
git show HEAD -- file.txt
```
